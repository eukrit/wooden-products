What shipped in 0.9.1 — WPC Fence palette fix + configurator polish

WPC Fence — palette + hero

  • .swatch-chip now sets background-size: cover, background-position: center, background-repeat: no-repeat on the class; each inline swatch uses explicit background-color + background-image instead of the background shorthand. Fixes reports of "palette doesn't show wood grains" on some browsers.
  • Added <link rel="preload" as="image" href="images/141.jpg"> so the hero paints on the first frame.

Configurator — randomised grain, trim plank, dimensions

  • Per-plank random grain crop — each plank renders the manufacturer swatch through a nested <svg> viewport with a seeded random x/y offset against an oversized image (1.6× wider / 1.4× taller). Each board reads as a distinct extruded length instead of one repeating tile. Seed is derived from the current spec so the pattern is stable when you change unrelated fields.
  • Trim plank at top — leftover space above the last full plank is now filled with a trim plank (field-cut on site). Drawn as a full 148 mm plank extending above the infill top; the top rail is painted after the planks so the overhang clips naturally. A dashed amber cut-line + ✂ Trim plank — N mm visible · cut on site badge mark the trim.
  • Bay width + fence height dimension lines — architectural-style dimensions outside the posts (horizontal above for bay width, vertical on the right for height) with extension lines, arrowheads, and boxed labels.
  • Board count update — boardsPerBay rounds up to include the trim. Totals strip shows "Includes N trim plank(s) — field-cut on site". Request-a-Quote spec preview adds Boards / bay and Total boards with a trim breakdown so the procurement team orders the right count.
  • SVG layer order — bottom rail before planks, top rail after planks. Trim overhang is clipped cleanly.
  • Removed the Powered by Gemini eyebrow next to "Render in a scene" — the render is still Gemini-backed, the tag wasn't earning its pixels.

Earlier — 0.7.4 (Submit flow, Phase 4)

What shipped in 0.7.4 — Submit flow (Phase 4)

POST /order/<id>/submit runs a 6-step flow: validate → allocate SO-WD-2026-NNNN via atomic Firestore transaction → snapshot live FX onto every line → create Xero DRAFT invoice (with auto-created Items + customer fallback) → post Block Kit to #orders-wood-products → mark submitted. Partial failures return HTTP 202 with {degraded:true}; order stays draft for admin retry.

Degraded-path logic

  • Xero creds missing → skip invoice, still post Slack (with "Xero draft pending" in context).
  • Xero call fails → order stays draft; response is HTTP 202 + {xero_error: "..."}.
  • Xero customer lookup 404 → auto-fallback to cached "Wood Product Customer" contact (cached once in counters/xero_fallback_contact_id).
  • Slack call fails → order stays draft; response is HTTP 202 + {slack_error: "..."}.
  • Both fail → order stays draft with both errors persisted on the doc for retry.
